## ✅ 正解說明
### 為什麼 B 是正確答案？

**Workspace Jobs UI** 是圖形化介面，需要**手動點擊操作**，無法用於程式化觸發。

#### 觸發 Job 的方式比較
| 工具 | 程式化觸發 | 說明 |
|------|-----------|------|
| **REST API** | ✅ 可以 | 透過 HTTP 請求觸發 |
| **Databricks CLI** | ✅ 可以 | 透過命令列觸發 |
| **Databricks SDKs** | ✅ 可以 | 透過 Python/Java/Go SDK 觸發 |
| **Workspace Jobs UI** | ❌ 不行 | 需要手動點擊 |

#### 程式化觸發範例
```python
# REST API
POST /api/2.1/jobs/run-now
{
  "job_id": 123
}

# Databricks CLI
databricks jobs run-now --job-id 123

# Python SDK
from databricks.sdk import WorkspaceClient
w = WorkspaceClient()
w.jobs.run_now(job_id=123)
```

#### UI vs 程式化
```
Workspace Jobs UI：
├── 需要登入 Web 介面
├── 手動點擊「Run Now」按鈕
├── 無法自動化或排程外部觸發
└── 適合手動測試和除錯

程式化工具（API/CLI/SDK）：
├── 可以整合到 CI/CD Pipeline
├── 可以從外部系統觸發
├── 可以寫腳本批次處理
└── 適合生產環境自動化
```

## ❌ 錯誤選項排除

### A. Databricks SDKs
**為什麼錯誤？**
- SDK 提供程式化觸發能力
- Python SDK: `w.jobs.run_now(job_id=123)`
- Java SDK: `workspace.jobs().runNow(123)`
- 這是程式化觸發的方式

### C. REST API
**為什麼錯誤？**
- REST API 是程式化觸發的**核心方式**
- 所有其他工具（SDK、CLI）底層都是呼叫 REST API
- `POST /api/2.1/jobs/run-now`

### D. Command-line interface (CLI)
**為什麼錯誤？**
- CLI 可以在腳本中使用
- `databricks jobs run-now --job-id 123`
- 完全支援程式化觸發

## 🧠 記憶法
### 口訣
**「UI 要手動，API/CLI/SDK 自動跑」**

### Job 觸發方式
```
Job 觸發方式：
├── 手動觸發
│   └── Workspace Jobs UI（點擊按鈕）
├── 程式化觸發
│   ├── REST API（HTTP 請求）
│   ├── CLI（命令列）
│   └── SDK（程式碼）
└── 排程觸發
    └── Scheduled Jobs（定時執行）
```

### 常見整合場景
| 場景 | 建議工具 |
|------|----------|
| CI/CD Pipeline | REST API / CLI |
| Python 應用程式 | Databricks SDK |
| Shell 腳本 | CLI |
| 手動測試 | Workspace Jobs UI |

### 注意「NOT」題型
```
反向題技巧：
├── 看到 "NOT", "EXCEPT", "Which is incorrect"
├── 標記這是反向題
├── 找出「唯一不符合」的選項
└── 其他選項都應該是「符合」的
```
