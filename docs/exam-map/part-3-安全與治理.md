7.數據安全與合規（Ensuring Data Security and Compliance，10%）

考察如何在 Databricks 保障資料安全和遵循法規要求。

也就是如何控制資料的存取權限、防止未授權使用，以及確保符合隱私保護規範（例如 GDPR）。涵蓋從存取控制、敏感資料保護，到資料留存和銷毀策略等主題。

範圍：

包括 Databricks 平臺的存取控制機制和資料保護措施。

考題可能涉及在工作區中設定物件的 ACL（存取控制清單）來實現最小權限原則，例如只給予使用者執行其職責所需的最低權限

涉及 Unity Catalog 提供的資料級安全功能，如列層級存取控制（Row-level filters）和欄位遮罩（Column masks），用以對敏感資料進行過濾或加密顯示

對合規性而言，範圍涵蓋資料匿名化/假名化技術（散列、代碼替換、遮蔽、泛化），以及如何在批次或串流管道中自動識別並遮蔽 PII（個人識別資訊）

資料留存政策：需要知道如何實現資料清除（purging）機制，確保在規定時間後刪除資料以符合法規

需要的知識：

了解 Databricks Workspace 和 Unity Catalog 的權限模型，尤其是 Unity Catalog 中目錄、資料庫(schema)、表之間的繼承關係

熟悉如何為 Tables/View 設定 ACL，以及如何透過 組和使用者管理許可。

掌握 Row Filter 和 Column Mask 的語法與用途，例如根據使用者角色自動套用某些資料篩選或遮蓋敏感欄位。

資料隱私保護，要知道各種遮蔽技術的效果：如 Hashing 可以不可逆地隱藏敏感值、Tokenization 可以用替代值代替真值且能對回原值進行控制、Generalization 通常用於模糊細節（例如只保留出生年不顯示日期）。

對 GDPR 等法規基本認識，例如「被遺忘權」能從備份和資料湖中刪除個人資料。

準備方式：

在 Databricks 上實踐設定幾種安全策略。

例如，使用 Unity Catalog 建立幾個資料庫和表，為不同使用者設定不同級別的存取權限，觀察權限繼承的情況。

嘗試為某個敏感資料表設置 Row Filter（如僅允許業務部門看自己部門的資料）和 Column Mask（如遮蔽身份證字號中間幾碼）。

編寫一個簡單的資料處理腳本，模擬檢測到 PII 資料（比如包含 email 或電話號碼的欄位）時自動將其轉換（如用 SHA2 函數雜湊）後再寫出，以體會如何在管道中落實匿名化。查閱 Delta Lake 對 時間旅行（Time Travel）和 VACUUM 的說明，理解如何永久刪除舊版本資料以滿足合規要求。

考試注意事項：

安全類題目常考對特定需求使用適當工具的匹配。

若題目要求「不同部門只能看自己資料」，應聯想到 Row Level Security

要求「遮蔽敏感欄位」，則是 Column Mask 或動態視圖。

看到敏感資訊或隱私字眼，要想到 GDPR 等規範，答案可能涉及資料匿名化或定期刪除

Databricks 跟雲端本身的安全集成，例如儲存區的 IAM 權限並不在考試重點內，重點是 Databricks 自身提供的控制。對於 Unity Catalog 權限繼承的考題，謹記上層權限傳遞給下層的規則，並理解Grant/revoke時可能的覆蓋情況。

8.數據治理（Data Governance，7%）

聚焦於管理和組織資料的制度與實踐。

資料治理確保企業資料有一致的定義、清晰擁有權和存取控制，以及良好的文件與註解，使資料易於發現且可信。

如何在技術層面和政策層面管好資料這份資產。

範圍：

資料目錄與中繼資料管理，以及 權限繼承模型。例如，如何在 Unity Catalog 中為資料資產添加描述或標籤，使得使用者容易搜尋到所需資料。

考題可能會問如何為一張表添加註解描述（讓別人明白其用途），或如何透過標籤（Tags）來標記資料的敏感等級。

Unity Catalog 權限繼承是範圍之一：例如當你在 Catalog 層級賦予某用戶使用權時，該使用權對下層所有 schema 和表的影響。

理解 目錄→模式→表 不同層級的 Grant 是如何運作的，可能會以情境題出現。

需要的知識：

熟悉 Unity Catalog 的結構：包含 Catalog、Schema、Table/View 等層次，以及在這些層次上可以存放與管理的中繼資料（例如描述、Owner、Tag）。

了解如何使用 SQL DDL 語句（如 **COMMENT ON TABLE**）或介面來添加資料的描述性資訊，幫助資料探索。

理解 權限繼承 模式，例如在 Catalog 層給一個群組 SELECT 權限意味著什麼、需要額外在表層級設定嗎 … 等

知道 Unity Catalog 與傳統 Hive Metastore 在治理上的差異（如 Unity Catalog 有細粒度許可且集中管理）。

準備方式：

在 Unity Catalog 中建立假想的資料，練習為它們添加 註解 和 屬性。

例如，建立一個銷售資料表，為其加上「此表包含公司年度銷售數據」的描述，並設定一個分類標籤如「敏感等級: 低」。

為不同使用者/群組授權，觀察在 Catalog 層級 vs Schema 層級賦權的效果差異。

閱讀關於 Unity Catalog 權限模型的章節，確保理解預設情況下，某使用者若無特定許可將無法看到目錄/資料庫的物件。

考試注意事項：

資料治理題不會非常技術細節，而是考你的理解和判斷。

例如可能描述一情境：「資料管理員希望確保所有表都有清楚的說明，以便其他團隊可以快速了解資料含義」，問你最佳做法——應該是使用資料目錄為表設置描述，而非發送郵件告知。或者問「若一使用者能查詢某張表，但無法看到同一Schema下的另一張表，可能原因是什麼」，考你對權限繼承/設定的理解。解題時，聯想 Unity Catalog 的層級結構和差異化設定即可。
